## 1. 개요

### 1.1 목적

반도체 시료(Sample)의 생산주문을 접수·승인·생산·출고까지 일관되게 추적·관리하는 콘솔 기반 생산주문관리 시스템.

### 1.2 배경

- 고객이 주문한 시료를 주문 담당자가 시스템에 등록하고, 생산 담당자가 재고 상태를 판단하여 직접 출고하거나 생산 후 출고하는 흐름을 단일 시스템에서 추적해야 한다.
- 시료별 재고·수율 정보를 바탕으로 부족분과 생산 소요 시간을 자동 산출하여 수작업 오류를 줄인다.

### 1.3 범위

| 포함 | 제외 |
|------|------|
| 시료 등록·조회·검색 | GUI 인터페이스 |
| 주문 접수·승인·거절 | 외부 ERP 연동 |
| 생산 계획 자동 산출 | 실시간 공정 모니터링 |
| 출고 처리 | 사용자 인증·권한 관리 |
| 콘솔 모니터링 (Rejected 제외) | 데이터베이스(RDBMS/NoSQL) |
| JSON 파일 기반 데이터 영속성 | 고객 직접 접속 기능 |

## 4. 비즈니스 흐름

### 4.1 주문 상태 전이

```
[고객 주문 의뢰]
      │
      ▼
 주문 담당자: 시스템 등록
      │
      ▼
 Reserved ──── 생산 담당자: 거절 ────► Rejected
      │                                 (모니터링 제외)
      │ 생산 담당자: 승인
      │
      ├── 재고 충분 ───────────────────► Confirmed
      │                                      │
      └── 재고 부족 ──► Producing             │ 생산 담당자: 출고 처리
                            │                 │
                            └── 생산 완료 ──► │
                                              ▼
                                           Released
```

### 4.2 상태 코드

| 값 | 상태 | 설명 | 전이 주체 |
|----|------|------|-----------|
| `0` | Reserved | 주문 접수 완료, 생산 담당자 검토 대기 | 주문 담당자 (등록) |
| `1` | Rejected | 생산 담당자가 주문 거절 | 생산 담당자 |
| `2` | Producing | 재고 부족으로 생산 대기 Queue 진입 | 시스템 자동 (승인 시 재고 부족 판정) |
| `3` | Confirmed | 재고 충분(즉시) 또는 생산 완료로 출고 가능 상태 | 시스템 자동 |
| `4` | Released | 출고 완료 | 생산 담당자 |

### 4.3 재고 판정 기준

#### 운영 맥락

- `current_stock`은 아직 출하되지 않은 **실물 재고**를 나타낸다.
- 재고 변동은 두 시점에만 발생한다.
  - **증가**: `Producing → Confirmed` 전환(생산 완료) 시 `shortage`만큼 증가한다.
  - **차감**: `Released` 전환 시 `order_quantity`만큼 감소한다.
- `Confirmed` 주문은 출하 대기 중이므로 `current_stock`에서 아직 차감되지 않았지만 해당 수량을 **전량 선점**하고 있다.
- `Producing` 주문은 부족분(`shortage`)을 생산으로 충당하고, 나머지(`order_quantity − shortage`)는 기존 재고에서 이미 **부분 선점**하고 있다.

#### 가용 재고 계산 (새 주문 승인 판정 시)

```
가용 재고 = current_stock
          − Σ(order_quantity | 동일 sample_id, 상태 = Confirmed)
```

| 상태 | 재고 선점 여부 | 판정 시 처리 |
|------|--------------|-------------|
| `Reserved` | 없음 | 미승인이므로 제외 |
| `Producing` | 첫 번째 주문이 가용 재고 전량 소진 | **가용 재고가 이미 0** → 판정 공식에서 별도 차감 불필요 (Case 2로 처리) |
| `Confirmed` | 전량 선점 | `order_quantity` 차감 |
| `Released` | 해당 없음 | 출하 시 이미 `current_stock`에서 차감됨 |

> `Producing` 주문의 선점분을 공식에 별도로 더하지 않아도 되는 이유: 첫 `Producing` 주문 이후에는 가용 재고가 항상 0이며, 이후 주문은 Case 2 분기(전량 생산)로 처리하기 때문이다.

#### 판정

```
가용 재고 ≥ 이번 주문의 order_quantity  →  Confirmed (재고 충분)
가용 재고 <  이번 주문의 order_quantity  →  Producing (재고 부족, 생산 레코드 자동 생성)
```

#### Producing 전환 시 부족분(shortage) 계산

가용 재고는 단 한 번만 소비될 수 있으므로, 동일 시료에 대해 이미 `Producing` 주문이 존재하는지 여부에 따라 계산 방식이 분기된다.

**Case 1 — 동일 시료에 대한 첫 Producing 주문**

```
가용 재고 = current_stock
          − Σ(order_quantity           | 동일 sample_id, 상태 = Confirmed)
shortage  = order_quantity − 가용 재고
```

이 주문이 남은 가용 재고를 전량 소진한다.

**Case 2 — 동일 시료에 대한 Producing 주문이 이미 존재**

```
shortage = order_quantity   (전량 생산)
```

앞선 `Producing` 주문이 이미 가용 재고를 전부 가져갔으므로, 이후 같은 시료의 `Producing` 주문은 재고 기여분이 없다. 주문량 전체를 생산으로 충당한다.

> **이 분기 로직의 목적**: 동시 또는 순차 승인 시 동일 가용 재고가 여러 주문에 중복 할당되는 사태를 방지한다.

## 5. 메인 메뉴 구조

주문 접수(주문 담당자)와 주문 처리(생산 담당자)는 역할이 완전히 달라 향후 역할별 분리를 고려해 메인 메뉴 레벨에서 독립 항목으로 구성한다.

```
[ 메인 메뉴 ]
 1. 시료 관리
 2. 주문 접수
 3. 주문 처리
 4. 모니터링
 5. 출고 처리
 6. 생산 라인
 0. 종료
```

### 메뉴별 목적 요약

| 번호 | 메뉴명 | 주 사용자 | 목적 |
|------|--------|----------|------|
| 1 | 시료 관리 | 주문 담당자 | 시료 등록, 목록 조회, 이름 검색 |
| 2 | 주문 접수 | 주문 담당자 | 고객 주문을 시스템에 등록 (`Reserved`) |
| 3 | 주문 처리 | 생산 담당자 | Reserved 주문 목록 조회 후 승인 또는 거절 |
| 4 | 모니터링 | 전체 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 5 | 출고 처리 | 생산 담당자 | Confirmed 주문 출고 실행 |
| 6 | 생산 라인 | 생산 담당자 | 현재 생산 중인 시료 및 대기 중 생산 큐 확인 |

### 6.2 주문 접수 (메뉴 2)

> **제약**: 주문은 `시료 관리`(메뉴 1)에 등록된 시료에 한해서만 가능하다.

#### 처리 흐름

```
입력 단계
 └─ 시료 ID / 고객명 / 주문수량 입력
       │
       ▼
확인 단계 (입력 내용 요약 표시)
 ┌─────────────────────────┐
 │  시료: 시료명 (시료ID)  │
 │  고객: 고객명           │
 │  수량: NNN ea           │
 └─────────────────────────┘
       │
  접수 / 취소
       │
  ┌────┴────┐
접수        취소 → 메뉴로 복귀
  │
  ▼
주문번호 발행 + 상태 = Reserved
결과 화면 표시
 ┌─────────────────────────────────┐
 │  주문번호: ORD-YYYYMMDD-NNN    │
 │  상태: Reserved                │
 └─────────────────────────────────┘
```

#### 요구사항

| ID | 요구사항 | 역할 | 우선순위 |
|----|----------|------|----------|
| FR-O-01 | 시료 ID, 고객명, 주문수량을 입력받는다. | 주문 담당자 | Must |
| FR-O-02 | 존재하지 않는 시료 ID 입력 시 오류 메시지를 출력하고 재입력을 유도한다. | 시스템 | Must |
| FR-O-03 | 입력 완료 후 `시료: 시료명 (시료ID) / 고객: 고객명 / 수량: N ea` 형태로 입력 내용을 요약 표시하고 접수 또는 취소를 선택받는다. | 시스템 | Must |
| FR-O-04 | 취소를 선택하면 주문을 등록하지 않고 메뉴로 복귀한다. | 시스템 | Must |
| FR-O-05 | 접수를 선택하면 주문번호(`ORD-YYYYMMDD-NNN`)를 자동 발행하고 상태를 `Reserved`로 저장한다. | 시스템 | Must |
| FR-O-06 | 접수 완료 후 발행된 주문번호와 현재 상태(`Reserved`)를 화면에 출력한다. | 시스템 | Must |

### 6.3 주문 처리 — 승인·거절 (메뉴 3)

#### 처리 흐름

```
Reserved 주문 목록 표시
 ┌────┬──────────────────────┬──────────┬────────────┬──────────┐
 │ No │ 주문번호             │ 고객     │ 시료       │ 수량     │
 └────┴──────────────────────┴──────────┴────────────┴──────────┘
       │
  번호(index)로 주문 선택
       │
       ▼
재고 확인 결과 표시
 ┌──────────────────────────────────────────────┐
 │  시료    : 시료명                            │
 │  주문수량: N ea                              │
 │  현재재고: N ea  (가용 재고, 선점분 제외)    │
 │  부족분  : N ea                              │
 │  [재고 부족 시 추가 표시]                    │
 │  실 생산량  : N ea                           │
 │  생산 소요  : HH:MM                         │
 └──────────────────────────────────────────────┘
       │
  승인 / 거절 / 취소(cancel)
       │
  ┌────┼────┐
승인  거절  취소 → 목록으로 복귀
  │    │
  ▼    ▼
주문번호 + 변경된 상태 표시
```

#### 요구사항

| ID | 요구사항 | 역할 | 우선순위 |
|----|----------|------|----------|
| FR-A-01 | `Reserved` 상태 주문 목록을 번호(index), 주문번호, 고객, 시료, 수량 컬럼의 테이블로 표시한다. | 시스템 | Must |
| FR-A-02 | 번호(index)를 입력하여 처리할 주문을 선택한다. | 생산 담당자 | Must |
| FR-A-03 | 선택된 주문에 대해 재고 확인을 수행하고 `시료명 / 주문수량 / 현재(가용)재고 / 부족분`을 표시한다. 가용 재고는 섹션 4.3의 판정 기준에 따라 산출한다. | 시스템 | Must |
| FR-A-04 | 부족분이 0보다 큰 경우(재고 부족), 실 생산량과 생산 소요 시간을 추가로 표시한다. | 시스템 | Must |
| FR-A-05 | 승인 / 거절 / 취소(cancel) 중 하나를 선택받는다. | 생산 담당자 | Must |
| FR-A-06 | 취소를 선택하면 주문 상태를 변경하지 않고 목록으로 복귀한다. | 시스템 | Must |
| FR-A-07 | 승인을 선택하면 재고 판정 결과에 따라 상태를 `Confirmed` 또는 `Producing`으로 자동 전환한다. `Producing` 전환 시 생산 레코드를 자동 생성하며, 이 시점에 `shortage`, 실 생산량, 소요 시간, 생산 시작 시각(`production_start_at`)을 모두 확정·저장한다. | 시스템 | Must |

### 6.6 생산 라인 (메뉴 6)

#### 개요

- `Producing` 주문은 FIFO 큐로 관리되며 한 번에 하나의 주문만 생산된다.
- 큐 진입(enqueue) 시점은 주문 승인(`Producing` 전환) 시각이며, 이 시점에 `shortage`, 실제 생산량, 소요 시간, **생산 시작 시각(`production_start_at`)**이 모두 확정·저장된다.
- `production_start_at` 계산 규칙: 큐가 비어있으면 `approved_at`, 선행 주문이 있으면 큐 마지막 주문의 `production_start_at + estimated_completion`으로 결정된다. 이로써 모든 대기 주문의 시간 정보가 enqueue 시점에 고정된다.
- 화면은 **생산 현황**(큐 최선두)과 **대기 주문**(큐 2번째 이후) 두 영역으로 구성된다.

#### 화면 구성

```
[ 생산 현황 ]
  주문번호    : ORD-YYYYMMDD-NNN
  시료명      : ...
  주문량      : N ea
  기존 가용재고: N ea        ← order_quantity − shortage
  부족분      : N ea
  실 생산량   : N ea
  수율        : NN %
  총생산시간  : HH:MM
  진행률      : [████████░░░░░░░░] NN %
  완료 예정   : HH:MM (+N day(s))

[ 대기 주문 ]
 ┌────┬──────────────────────┬──────────┬──────┬──────┬──────────┬─────────────────────┐
 │순서│ 주문번호             │ 시료명   │주문량│부족분│실 생산량 │예상 완료시간        │
 ├────┼──────────────────────┼──────────┼──────┼──────┼──────────┼─────────────────────┤
 │  1 │ ORD-YYYYMMDD-NNN    │ ...      │  N   │  N   │  N       │ HH:MM               │
 │  2 │ ORD-YYYYMMDD-NNN    │ ...      │  N   │  N   │  N       │ HH:MM (+1 day)      │
 │  3 │ ORD-YYYYMMDD-NNN    │ ...      │  N   │  N   │  N       │ HH:MM (+2 days)     │
 └────┴──────────────────────┴──────────┴──────┴──────┴──────────┴─────────────────────┘
```

> 완료 예정·예상 완료시간 표시 기준: 화면 조회 시각의 날짜와 동일한 경우 `HH:MM`, 다음 날 이후면 `HH:MM (+N day(s))` 형식. 생산 현황과 대기 주문 모두 동일 규칙을 적용한다.

#### 요구사항

| ID | 요구사항 | 역할 | 우선순위 |
|----|----------|------|----------|
| FR-L-01 | 생산 현황 영역에 큐 최선두 주문의 주문번호, 시료명, 주문량, 기존 가용재고(`order_quantity − shortage`), 부족분, 실 생산량, 수율, 총생산시간을 표시한다. | 시스템 | Must |

| FR-L-04 | 큐 정렬 기준은 주문 승인(`Producing` 전환) 시각이며, 먼저 승인된 주문이 높은 우선순위를 갖는다. | 시스템 | Must |

| FR-L-09 | 생산 중인 주문의 완료 예정 시각(`production_start_at` + `estimated_completion`)이 경과하면 백그라운드에서 자동으로 해당 주문을 `Confirmed`로 전환하고 해당 시료의 `current_stock`을 `shortage`만큼 증가시킨다. `production_start_at`은 enqueue 시점에 이미 확정되어 있으므로 전환 시 별도 갱신이 불필요하다. | 시스템 | Must |

> **데이터 스키마 변경 필요**: 생산 라인 기능 구현을 위해 `Order` 레코드에 승인 시각(큐 진입 기준 정렬용) 및 진행률 계산에 필요한 생산 시작 시각 등 추가 필드가 필요하다. [`lib/persistence/DATA_SCHEMA.md`](../lib/persistence/DATA_SCHEMA.md) 변경이 수반된다.
